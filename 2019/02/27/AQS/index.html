抽象队列式同步器

发表于 2019-02-27 | 分类于 多线程 | | 阅读次数:

AbstractQueuedSynchronizer(AQS)，抽象的队列式同步器，AQS定义了一套多线程访问共享资源的同步器框架，许多同步类实现都依赖于它，如常用的ReentrantLock/Semaphore/CountDownLatch等。

AQS设计：

  • 使用Node实现FIFO队列，可以用于构建锁或者其他同步装置的基础框架
  • 利用了一个int类型state表示状态。如ReentrantLock中的state，state = 0表示还没有线程获得锁，state = 1表示已经有线程获得锁，state > 1表示可重入锁的数量
  • 使用方法是继承
  • 子类通过继承并通过实现它的方法管理其状态，通过acquire和release方法操作状态
  • 可以同时实现怕排他锁和共享锁模式（独占、共享）

AQS同步组件

  • CountDownLatch
  • Semaphore
  • CyclicBarrier
  • ReentrantLock
  • Condition
  • FutureTask

ConuntDownLatch

CountDownLatch有一个计数器，当一个线程调用了await()方法之后，该线程会被阻塞，其他线程如果调用countDown()方法，计数器的值减1，当计数器的值减为0时，原来的阻塞线程会被唤醒并继续执行。

CountDownLatch适合于一个任务的执行需要依赖多个子任务的执行结果，这时可以考虑使用CountDownLatch。

Semaphore

Semaphore用于控制同时执行的线程数，使用acquire()方法获取许可，如果没有则阻塞，通过release()方法释放许可。

Semaphore(int permits):构造方法，创建具有给定许可数的计数信号量并设置为非公平信号量。

Semaphore(int permits,boolean fair):构造方法，当fair等于true时，创建具有给定许可数的计数信号量并设置为公平信号量。

acquire()和release()方法可以添加参数，表示同时获取/释放多个许可。

Semaphore的tryAcquire()方法，表示尝试获取许可，可以传参

表示尝试等待timeout的时间来获取许可，若获取不到则结束运行，避免程序长时间等待。

CyclicBarrier

CyclicBarrier也有一个计数器，但是与CountDownLatch不同的是，这个计数器是向上增加的。

ReentrantLock(可重入锁)

在Java中通常实现锁有两种方式，一种是synchronized关键字，另一种是Lock。

区别：
synchronized是基于JVM层面实现的，而Lock是基于JDK层面实现的。

使用：
对于使用者的直观体验上Lock是比较复杂的，需要lock和realse，如果忘记释放锁就会产生死锁的问题，所以，通常需要在finally中进行锁的释放。但是synchronized的使用十分简单，只需要对自己的方法或者关注的同步对象或类使用synchronized关键字即可。但是对于锁的粒度控制比较粗，同时对于实现一些锁的状态的转移比较困难。

在JDK1.5之后synchronized引入了偏向锁，轻量级锁和重量级锁，从而大大的提高了synchronized的性能。

ReentrantLock是实现Lock接口的一个类，支持重入性，表示能够对共享资源能够重复加锁，即当前线程获取该锁再次获取不会被阻塞。ReentrantLock还支持公平锁和非公平锁两种方式。

公平锁：每次获取到锁为同步队列中的第一个节点，保证请求资源时间上的绝对顺序

公平锁每次都是从同步队列中的第一个节点获取到锁，
而非公平性锁则不一定，有可能刚释放锁的线程能再次获取到锁。

重入性实现

非公平锁尝试锁定线程时，先判断当前锁的状态，若未被占用，则尝试获取该锁，compareAndSetState方法通过调用Unsafe类的CAS算法实现。若当前锁被占用，查看是否为当前线程，若是则state+1，表示重入。

释放锁时先把同步状态减1，判断持有锁的线程是否是当前线程，不是的话抛出异常。然后再判断锁是否被释放。加锁之后相当于单线程运行，此时不需要使用CAS来给state赋值。

与非公平锁相比，公平锁在获取锁时需要判断当前线程是否是等待队列里等待最久的线程。

公平锁与非公平锁比较：

公平锁是严格的以FIFO的方式进行锁的竞争，但是非公平锁是无序的锁竞争，刚释放锁的线程很大程度上能比较快的获取到锁，队列中的线程只能等待，所以非公平锁可能会有“饥饿”的问题。但是重复的锁获取能减小线程之间的切换，而公平锁则是严格的线程切换，这样对操作系统的影响是比较大的，所以非公平锁的吞吐量是大于公平锁的，这也是为什么JDK将非公平锁作为默认的实现。

Callable和Future

创建线程有两种方式，一是继承Thread类，二是实现Runnable接口。

但是这两种实现方式都存在问题，那就是不能获取线程运行的结果，若要获取运行结果就要通过线程通信或共享变量的方式，实现起来比较麻烦。

Jdk1.5之后，提供了Callable和Future两个接口，可以获取线程的运行结果。
